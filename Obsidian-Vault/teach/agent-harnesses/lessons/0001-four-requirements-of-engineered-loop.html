Lesson 001 — Foundations

The Four Requirements of an Engineered Loop

Every loop that ships real work — not just generated noise — rests on exactly four pillars.

Before you build a harness, you need to understand what makes a loop engineered rather than accidental. A loop with no structure is a "slop cannon": it spins, generates, and produces noise. An engineered loop has constraints that turn non-deterministic generation into deterministic progress.

There are exactly four requirements. Missing any one collapses the loop into randomness.

1. A Goal

The goal scopes the loop. It answers: what does "done" look like?

Without a goal, the loop has no exit condition and no success metric. The agent spirals because "better code" is not a boundary — it's a direction with no destination.

Examples of well-scoped goals:

# ✅ Good
"Fix the ClickHouse query latency on the events table"
"Get CI green on the auth branch"
"Implement all items marked TODO in fix_plan.md"

# ❌ Bad
"Make the codebase better"
"Write more tests"
"Improve performance"
Insight

PostHog's loops team found that the most effective goals are binary and measurable: CI green or not, test passes or fails, latency under X ms or not. Binary goals eliminate the agent's ability to "feel done" without evidence.

2. Context

Context is the fuel the agent pulls from during each iteration. The critical distinction: context is curated and fed throughout the run, not dumped upfront.

Good context includes:

  • Specifications (requirements as files)
  • Current plan / TODO list
  • Recent errors and failures
  • Build/run instructions
  • Skill files the agent can consult
  • Memory of previous iterations

The agent should fetch and react to new inputs — not work from a static dump. In the Ralph technique, this means reading fix_plan.md and specs/ fresh each iteration. In the Missions architecture, workers inherit a clean slate plus a structured handoff from the previous worker.

Pitfall

Dumping the entire codebase into the context window is the most common mistake. The context window is ~170k tokens effective — and the agent can only work with what it attends to. Curate ruthlessly. The agent should request context, not be forced to ignore it.

3. Evaluation

Evaluation is how the agent checks itself. This is the key distinction from standard prompting: in a loop, the agent does the verification, not the engineer.

Common evaluation mechanisms:

  • Tests — unit, integration, end-to-end
  • Benchmarks — for performance loops (autoresearch)
  • CI pipelines — for integration-level gates
  • LLM-as-judge — the LLM scores its own output against criteria
  • Static analysis — type checkers, linters, Dialyzer, Pyrefly
  • User testing validators — spawn the app, interact via computer use

The evaluation criterion should be defined before implementation. In the Missions architecture, this is the validation contract — hundreds of assertions written during planning, before any code exists.

Key insight

Tests written after implementation confirm decisions rather than catch bugs. Validation contracts prevent this by defining correctness independently of code. The difference: "this function returns a user object" (post-hoc) vs "the API must return a 200 with a 'id' field and 'name' field" (pre-contract).

4. An Agent

The agent is the actor. It can range from a simple while true; do claude-code; done loop to a sophisticated multi-role system. What matters is not the sophistication of the agent, but that it has the tools, instructions, and autonomy to act on the other three requirements.

Agent architectures in the wild:

Architecture Agents Best for
Ralph 1 monolithic Greenfield, single developer
Missions 3 roles (orchestrator, workers, validators) Multi-day feature delivery
Karpathy autoresearch 1 + benchmark Performance optimization

The choice between monolithic and multi-agent is not about intelligence — it's about complexity management. Ralph's philosophy: "Multi-agent, agent-to-agent communication and multiplexing adds enormous complexity — especially when the agents themselves are non-deterministic." One agent, one loop, one repo is simpler and often more effective.

Bonus: The Validation Contract

While not one of the four requirements, the validation contract is the most important design pattern inside engineered loops. It bridges the Goal and Evaluation requirements:

  1. Written during planning, before any code
  2. Defines correctness independently of implementation
  3. Each feature gets one or more assertions it must satisfy
  4. Sum of all features means every assertion is covered
  5. Prevents tests from being "shaped by the code"

This is what Luke Alvoeiro at Factory calls the most critical innovation in multi-agent engineering: define done before you start building.
