# Synthetic Data vs Metrics in Evaluation

In AI evaluation, there are two distinct components that serve different purposes: **synthetic data** (what you test with) and **metrics** (how you judge results).

## Synthetic Data

Synthetic data is the test data used to evaluate a system. It answers the question: *what should we test?* Synthetic data generation involves creating realistic input scenarios that exercise the system's capabilities and edge cases. ^[extracted]

## Metrics

Metrics are the measurement criteria used to judge system outputs. They answer the question: *how do we know if the output is good?* Metrics define what quality looks like and how it is measured. ^[extracted]

## Why Distinguish Them

Treating synthetic data and metrics as separate concerns allows teams to:

1. **Build better test data** without being constrained by how it will be measured
2. **Iterate on metrics independently** of test data generation
3. **Reuse test data** across different metric configurations
4. **Debug failures more precisely** — knowing whether the problem is bad test data or bad metrics
## Synthetic Data in Robotics

In robotics, synthetic data plays a fundamentally different role than in AI evaluation: ^[extracted]

+ **Training data, not test data** — Synthetic simulation data forms the middle tier of the [[concepts/data-pyramid|Data Pyramid]] for training foundation models, not for evaluation
+ **Physics-based simulation** — Generated by engines like NVIDIA Omniverse running on OVX hardware, producing robot-environment interaction data
+ **Data multiplication** — [[concepts/dreamgen|DreamGen]] extends synthetic data by generating variants of teleoperation trajectories through world foundation models

The distinction between synthetic data (what you test with) and metrics (how you judge results) is especially important in robotics because the "test data" is also the "training data" — the same synthetic datasets that train the model are used to evaluate its capabilities. ^[inferred]
