# Semantic Memory Noise

**Semantic memory noise** is the problem where irrelevant facts generated by an LLM pollute agent memory stored in a vector database, causing retrieval to return unrelated facts and leading to inaccurate responses or hallucinations. ^[extracted]

## The Mechanism

When an agent generates facts during conversation and stores them as vectors:

1. **Free-form fact generation** produces facts across many domains (e.g., "I wake up at 7 AM", "my dog's name is Melody", "I listen to jazz music")
2. **All facts go into the same vector space** with no domain separation
3. **Retrieval by semantic similarity** returns facts that are close in embedding space, regardless of whether they are relevant to the current task
4. **Irrelevant facts in context** cause the LLM to produce inaccurate or hallucinated responses ^[extracted]

## Concrete Example

A media player agent should remember facts about music preferences (jazz, NPR podcasts, The Daily). But because the agent generates arbitrary facts during conversation, it also captures:
- "I wake up at 7 AM"
- "My dog's name is Melody"

When the agent searches for "my favorite tunes," it retrieves "Melody" because the word "Melody" is semantically similar to "tunes" — even though the dog's name has nothing to do with the music domain. ^[extracted]

## Root Cause

The root cause is that **vectors are projections into an embedding space with no causal or relational structure**. There is no mechanism to differentiate domain-relevant facts from irrelevant ones. Semantic similarity captures lexical and topical proximity, not business relevance. ^[extracted]

## Solutions

- **[[concepts/domain-aware-memory|Domain-aware memory]]** — Define entity types per domain; only those entities are stored and retrieved
- **[[concepts/graphrag|GraphRAG]]** — Structured graph storage with typed nodes enables filtered retrieval by entity type
- **[[concepts/ontology-driven-retrieval|Ontology-driven retrieval]]** — Ontology annotations control which relationships to traverse
