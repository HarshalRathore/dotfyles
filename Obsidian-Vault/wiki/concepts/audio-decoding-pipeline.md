# Audio Decoding Pipeline

The **audio decoding pipeline** is the component of a TTS inference system that converts discrete audio tokens (generated by the TTS model) into continuous waveform audio. It sits after the LLM-like next-token prediction and is a distinct optimization challenge from the text generation itself. ^[extracted]

## The Challenge

Unlike text LLMs where the output is already human-readable tokens, TTS produces discrete audio codes that must be decoded into audible waveform. This adds a non-trivial computation step:

1. **TTS model** generates discrete audio tokens (like laugh, pause, phoneme codes)
2. **Audio decoder** (e.g., Snac) converts these tokens into waveform samples
3. **Streaming** delivers audio to the client in real-time ^[extracted]

## GPU-Accelerated Decoding

Philip Kiely's approach at [[entities/baseten|Baseten]] applies the same compilation system used for model weights to the audio decoder:

- **Snac** — the audio decoder library (notably not named after a snack despite the name)
- **Torch compile** — compiles the decoder for optimized execution
- **PyTorch inference mode** — runs the decoder on GPU, not CPU
- This brings audio decoding into the GPU loop, reducing CPU-GPU transfer overhead ^[extracted]

## CPU-Bound Discovery

Surprisingly, profiling revealed that the TTS pipeline with Orpheus is often **CPU-bound** rather than GPU-bound. Both the next-token prediction loop and the audio decoding loop hit the CPU at different points in the pipeline. This means:

- Adding more GPUs doesn't help if CPU is the bottleneck
- CPU count, not GPU count, limits simultaneous streams
- This is actually favorable — CPUs scale more linearly and cheaply than GPUs ^[extracted]

## Batching Integration

Dynamic batching operates across the entire pipeline — token batching for the TTS model AND audio decoding batches are coordinated to maximize throughput while maintaining the 15ms flush timeout. ^[extracted]
