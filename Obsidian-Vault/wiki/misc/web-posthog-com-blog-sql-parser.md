# I wrote a 70x faster SQL parser while barely looking at the code — Robbie Coomber

> Robbie Coomber, PostHog. Jun 24, 2026. Originally at https://posthog.com/blog/sql-parser. Cross-posted to X as a long-form article.
> Also on Hacker News: https://news.ycombinator.com/item?id=48663544

## Overview

After the success of using agents to improve ClickHouse query performance through autoresearch, Robbie Coomber at PostHog attempted something more ambitious: rewriting their SQL parser entirely using AI agents. The old parser was generated by ANTLR (C++), a parser generator that compiles a `.g4` grammar file into a generic graph-walking interpreter. The new parser is a hand-rolled recursive-descent parser in Rust with a Pratt expression core, written entirely by Claude Opus 4.7 in May 2026. Result: **454x faster in production, 70x faster on benchmarks.**

## Architecture

### Old: ANTLR-generated C++
- Grammar defined in `.g4` file → ANTLR compiles into an ATN (NFA-with-a-stack)
- Generic graph-walking interpreter at runtime — flexible but slow
- Extremely powerful, handles complex grammars, but every token requires indirection through the ATN interpreter
- C++ but not fast C++ — the bottleneck is the interpreter architecture, not the language

### New: Hand-rolled Rust by Claude
- Predominantly recursive-descent parser with a Pratt expression core
- LL(2) cursor widened at specific spots by bounded non-consuming lookahead probes
- Localized ordered-choice speculative backtracking for the few decisions that need it
- Written entirely by Claude Opus 4.7, never directly edited by a human
- 16K lines of parser code + 5K lines of tooling + tests

## The Agent Loop

### Two parallel approaches tested simultaneously
1. **Performance-first** — recursive-descent with Pratt expression loop, minimal lookahead and backtracking
2. **Faithful-to-ANTLR** — followed ANTLR's behavior but implemented transitions in explicit code

Both approaches converged on similar quality. The existence of the oracle (existing C++ parser) was critical: **test-driven development by finding disagreements and fixing them.**

### The Iteration Loop

1. Generate new test failures from PBT (Hypothesis), real production corpus, regression tests, and "think really hard about edge cases" agent
2. Shrink (minimize) failing test cases using ShrinkRay for non-Hypothesis sources; Hypothesis does this automatically
3. Add shrunk failures to an expanding regression test suite
4. Load grammar file + relevant C++ source into context — then think about the best fix, preferring general solutions
5. Make the fix and print a one-paragraph summary for the human
6. Run regression suite
7. Re-run autonomously

### Property-Based Testing Stack
- **Hypothesis (Python)** — generates SQL queries that exercise the parser, finds disagreements
- **Codegen'd SQL generator** — wrote a tool to parse the ANTLR `.g4` grammar file and generate an SQL generator
- **Coverage-guided generation** — biases toward constructs not yet exercised
- **ShrinkRay** — minimizes failing test cases from non-Hypothesis sources
- **Production query log** — anonymized real queries as test cases
- The PBT runs constantly in the background, writing new failures to a file for Claude to fetch

### Key Prompt Engineering Insight
Claude kept making brittle fixes (one-token lookahead → needed two-token → context window forgot). The fix: load both the grammar file and relevant C++ source code into context immediately before writing any fix code. This prompt structure solved the "forgetting" problem.

## Results

!
*Benchmark results: new Rust parser vs old ANTLR C++ parser — 70x faster on laptop benchmarks, 454x faster on production queries.*

- **70x faster** on laptop benchmarks
- **454x faster** on production queries (longer SQL that didn't hit the parser cache)
- **Zero divergences** in shadow mode testing millions of production queries
- Switched over production traffic (with 0.1% reverse shadow) after a couple of hours of shadow mode

The performance gap exists because:
- ANTLR's generic graph-walking interpreter does a lot more work per token
- The hand-rolled parser uses explicit code paths with no indirection
- Rust's performance characteristics vs C++ (both fast, but the difference is architecture, not language)

## Open Questions

- How reproducible is this approach for non-parser domains? The oracle (existing C++ parser) was critical — not every rewrite target has one.
- Would this work for a greenfield parser without an oracle? The article explicitly says "having an oracle was critical."
- Does the Pratt expression + LL(2) cursor architecture generalize to more complex grammars (e.g. full SQL with recursive CTEs, window functions)?
